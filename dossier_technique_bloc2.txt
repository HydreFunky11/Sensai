--------------------------------------------------------------------------------
INTRODUCTION ET GENESE DU PROJET
--------------------------------------------------------------------------------

L'apprentissage d'une langue etrangere comme le japonais represente un defi unique,
particulierement en raison de la complexite de son systeme d'ecriture (compose de
Hiragana, Katakana et de milliers de Kanji). Les etudiants se tournent souvent vers
des medias de culture populaire, tels que les mangas, pour s'immerger dans la langue
reelle. Cependant, le processus de lecture de mangas physiques ou numeriques est
constamment interrompu par la necessite de rechercher des caracteres inconnus.
Les dictionnaires classiques necessitent de connaitre la cle du kanji, son nombre
de traits, ou d'utiliser un outil de dessin de caractere approximatif, ce qui brise
le rythme de lecture et reduit l'engagement de l'apprenant.

SensAI est ne de la volonte de resoudre cette problematique. Il s'agit d'une
application web d'apprentissage intelligente combinant un lecteur de manga
numerique, un outil de reconnaissance optique de caracteres (OCR) local et
specifique aux bulles de mangas japonais, un systeme de traduction assistee par
intelligence artificielle (LLM) qui decortique la grammaire, et un systeme de
repetition espacee (SRS) base sur l'algorithme SM-2 pour stocker et reviser le
vocabulaire sous forme de fiches interactives (flashcards).

Ce dossier detaille l'integralite de la conception technique et de l'assurance
qualite de SensAI dans le cadre du Bloc 2 de certification. Il aborde la
conception de l'architecture backend, la securisation des acces et des fichiers,
la conformite vis-a-vis du RGPD (droit a l'oubli et portabilite), les tests
d'integration et d'accessibilite (A11y), le plan de correction des bogues
critiques rencontres en cours de developpement, ainsi que les procedures de
deploiement et d'exploitation de l'infrastructure.

PARTIE I : ANALYSE ET CONCEPTION TECHNIQUE DE L'INFRASTRUCTURE BACKEND
================================================================================

--------------------------------------------------------------------------------
1.1 Choix des technologies et architecture generale
--------------------------------------------------------------------------------
L'application SensAI adopte une architecture découplée de type Single Page
Application (SPA) communiquant via une API REST avec un serveur backend.

Le backend est developpe en Python avec le framework FastAPI. Ce choix s'explique
par plusieurs facteurs clefs :
- Performance : FastAPI est l'un des frameworks web Python les plus rapides, rivalisant
  avec Node.js et Go grace a son support natif de la programmation asynchrone (async/await).
- Typage et validation : FastAPI s'appuie sur Pydantic pour la validation stricte des
  donnees en entree et en sortie. Cela permet de bloquer automatiquement les requetes
  malformees et de securiser les points d'entree de l'API.
- Documentation automatique : Il genere automatiquement la documentation interactive
  Swagger/OpenAPI, facilitant la collaboration et les tests.

La base de donnees choisie est SQLite, geree a travers l'ORM SQLAlchemy. Pour une
premiere version ciblee utilisateur individuel ou auto-hebergee (Self-hosted), SQLite
est un choix parfait :
- Fichier unique : Pas de base de donnees lourde a maintenir ou a configurer en reseau,
  ce qui simplifie enormement le deploiement et les sauvegardes.
- Vitesse : SQLite s'execute en memoire ou en acces direct sur le disque local du
  serveur, evitant les temps de latence reseau d'une BDD classique.
- Securite : Toutes les requetes passent par l'ORM SQLAlchemy sous forme de requetes
  parametrees, rendant l'application completement immunisee contre les injections SQL.

--------------------------------------------------------------------------------
1.2 Schema de base de donnees et cascades de suppression (RGPD)
--------------------------------------------------------------------------------
La base de donnees de SensAI est structuree de maniere a preserver l'integrite referentielle
tout en assurant la suppression totale des donnees d'un utilisateur lorsqu'il en
fait la demande (Droit a l'oubli - Article 17 du RGPD).

Voici la description des tables principales definies dans 'manga-ocr/db/models.py' :

*   Table 'users' (Utilisateurs) :
    - id (Integer, Cle primaire)
    - email (String, unique, indexe)
    - hashed_password (String) : Stocke le mot de passe chiffre par l'algorithme bcrypt.
    - stripe_customer_id (String, unique, optionnel)
    - is_premium (Boolean)

*   Table 'manga_folders' (Dossiers de la Bibliotheque) :
    - id (Integer, Cle primaire)
    - user_id (Integer, Cle etrangere pointant vers 'users.id')
    - name (String)

*   Table 'mangas' (Livres et chapitres importes) :
    - id (Integer, Cle primaire)
    - user_id (Integer, Cle etrangere pointant vers 'users.id')
    - folder_id (Integer, Cle etrangere pointant vers 'manga_folders.id')
    - title (String)
    - file_path (String) : Chemin vers le fichier image ou PDF sur le disque.
    - cover_image (String) : Miniature codee en Base64.

*   Table 'decks' (Dossiers de revisions SRS) :
    - id (Integer, Cle primaire)
    - user_id (Integer, Cle etrangere pointant vers 'users.id')
    - title (String)
    - description (String)

*   Table 'flashcards' (Cartes de revision de vocabulaire) :
    - id (Integer, Cle primaire)
    - deck_id (Integer, Cle etrangere pointant vers 'decks.id')
    - text_source (String) : Texte japonais capture par l'OCR.
    - translation (String) : Traduction francaise du mot ou de la phrase.
    - romaji (String) : Prononciation phonetique.
    - breakdown (JSON) : Decoupage lexical mot par mot.
    - context_note (String) : Note ou type de carte ('character' ou 'word').

*   Table 'review_stats' (Statistiques d'apprentissage SM-2) :
    - id (Integer, Cle primaire)
    - flashcard_id (Integer, Cle etrangere pointant vers 'flashcards.id')
    - next_review_date (DateTime) : Prochaine date de revision calculee.
    - interval (Integer) : Nombre de jours d'attente avant la prochaine revision.
    - ease_factor (Float) : Facteur d'aisance utilise par l'algorithme SM-2 (par defaut 2.5).

*   Table 'learned_characters' (Lettres japonaises apprises) :
    - id (Integer, Cle primaire)
    - user_id (Integer, Cle etrangere pointant vers 'users.id')
    - character (String)
    - alphabet_type (String) : 'hiragana', 'katakana' ou 'kanji'.

Afin d'eviter des incoherences ou des violations de contraintes d'integrite de clés
etrangeres en base de donnees lors de la suppression d'un utilisateur, SQLAlchemy a
ete configure avec des regles de cascades precises :
```python
# Extrait du fichier models.py :
class User(Base):
    __tablename__ = "users"
    # ...
    mangas = relationship("Manga", back_populates="owner", cascade="all, delete-orphan")
    decks = relationship("Deck", back_populates="owner", cascade="all, delete-orphan")
    learned_characters = relationship("LearnedCharacter", back_populates="owner", cascade="all, delete-orphan")
    folders = relationship("MangaFolder", back_populates="owner", cascade="all, delete-orphan")
```
Grace a cette configuration, lorsqu'un appel de type 'db.delete(user)' est effectue :
1. Tous les enregistrements associes dans les tables 'mangas', 'decks', 'learned_characters'
   et 'manga_folders' sont automatiquement identifies et effaces de la base.
2. Pour les 'decks', la cascade se propage : la suppression d'un deck supprime toutes
   ses 'flashcards' liees.
3. Pour les 'flashcards', la cascade se propage a son tour vers la table 'review_stats'
   pour supprimer l'historique d'apprentissage correspondant.
Ce mécanisme garantit une purge automatique, instantanee et complete de la base de
donnees relationnelle, repondant parfaitement a la reglementation de l'Union Europeenne.

TEST ID : T-001 | SECURITE DE L'AUTHENTIFICATION & CHIFFREMENT
--------------------------------------------------------------------------------
- Description detaillee :
  Verifie que les mots de passe utilisateurs ne sont jamais stockes en clair sur
  le serveur et que le mecanisme d'enregistrement rejette les doublons d'adresse email.
- Pre-requis :
  Le serveur de base de donnees SQLite et le backend FastAPI sont initialises.
- Protocole de test :
  1. Envoyer une requete POST vers '/auth/register' avec le JSON suivant :
     {"email": "candidat@bloc2.fr", "password": "MonPasswordSuperSecurise123!"}
  2. Noter la reponse du serveur.
  3. Recommencer l'operation avec exactement le meme payload JSON.
  4. Ouvrir un terminal et executer une requete SQL de controle sur le serveur :
     sqlite3 sensai.db "SELECT hashed_password FROM users WHERE email='candidat@bloc2.fr';"
- Resultats observes :
  * Premier envoi : HTTP 200 renvoye. Un dictionnaire contenant 'access_token' (JWT)
    et 'token_type' ("bearer") est retourne.
  * Deuxieme envoi : Le serveur intercepte la duplication de l'email au niveau de
    la base de donnees et leve une exception HTTP 400 avec le detail "Email deja enregistre".
  * Requete SQL de controle : Le mot de passe stocke est un hash du type :
    "$2b$12$N9qo8F...". Il s'agit d'une signature chiffre asymetrique generee par
    bcrypt. Le mot de passe en clair "MonPasswordSuperSecurise123!" est introuvable.
- Conclusiveness : SUCCES

TEST ID : T-002 | VERIFICATION DES ENTETES HTTP DE SECURITE (OWASP)
--------------------------------------------------------------------------------
- Description detaillee :
  S'assurer que toutes les requetes sortant de l'API FastAPI contiennent les en-tetes

PARTIE III : PLAN DE CORRECTION DES BOGUES (REGISTRE DES ANOMALIES RESOLUES)
================================================================================

Cette section decrit les quatre dysfonctionnements majeurs identifies durant la
phase de construction et de test de SensAI, ainsi que les methodes de diagnostic
et de resolution adoptees.

--------------------------------------------------------------------------------
ANOMALIE ID : B-001 | COMPOSANT : Backend - Importation et traitement des mangas PDF
--------------------------------------------------------------------------------
*   Symptomes et description :
    Lorsqu'un utilisateur tentait d'uploader un tome complet de manga au format PDF
    (taille superieure a 50 Mo, contenant generalement plus de 150 pages de scans haute
    definition), le conteneur Docker du backend s'arretait brutalement sans renvoyer
    de reponse. Les outils de monitoring systeme indiquaient un arret de type "OOM"
    (Out Of Memory - Plus de memoire vive disponible).
*   Diagnostic et demarche de recherche :
    En surveillant la consommation de memoire vive du processus Python FastAPI avec
    la commande systeme 'top' lors de l'upload, nous avons constate une augmentation
    fulgurante de la RAM (depassant les 2 Go alloues au conteneur) juste apres la
    reception du fichier. La cause venait de l'utilisation de la bibliotheque de lecture
    PDF standard qui tentait de charger l'integralite de la structure du document et
    de ses images non compressees en memoire vive avant de proceder a la decoupe.
*   Resolution technique appliquee :
    Remplacement de la routine d'importation par l'integration de la bibliotheque
    PyMuPDF ('fitz'). Le code a ete reecrit pour ouvrir le document sous forme de flux
    direct (stream) depuis la memoire tampon de FastAPI.
    L'extraction de la plage demandee (ex: pages 1 a 12) s'effectue en selectionnant
    uniquement les index de pages du document d'origine et en les inserant dans un
    nouveau document vierge en memoire, reduisant considerablement la memoire requise
    (seuls 80 Mo sont desormais utilises, meme pour des documents de 200 Mo).
    Le fichier decoupe est ensuite ecrit sur le disque en une seule passe.
    Des validateurs stricts ont egalement ete integres dans 'core/security.py' pour
    rejeter immediatement tout fichier superieur a 100 Mo.

--------------------------------------------------------------------------------
ANOMALIE ID : B-002 | COMPOSANT : Frontend - Suite de tests unitaires d'onglets
--------------------------------------------------------------------------------
*   Symptomes et description :
    Suite au renforcement de l'accessibilite sur la page de selection des alphabets
    japonais (Hiragana, Katakana, Kanji), la commande de validation de pipeline 'npm
    run test' a commence a echouer systematiquement, bloquant l'integration continue
    sur GitHub Actions.
*   Diagnostic et demarche de recherche :
    L'erreur retournee par Vitest et Testing Library etait :
    "TestingLibraryElementError: Unable to find an element with the role: button and name: /hiragana/i"
    L'examen du code source de 'Alphabets.jsx' a revele que pour nous conformer aux
    directives WAI-ARIA d'accessibilite d'une liste d'onglets, nous avions modifie
    les balises '<button>' de navigation d'alphabets. Elles possedaient a present
    l'attribut 'role="tab"' au lieu du role de bouton HTML generique par defaut.
    L'arbre d'accessibilite s'etant mis a jour, l'outil de test ne parvenait plus
    a les trouver en cherchant des elements ayant le role "button".
*   Resolution technique appliquee :
    Modification du fichier de test 'Alphabets.test.jsx'.
    Les requetes d'onglets de la forme :
      expect(screen.getByRole('button', { name: /hiragana/i })).toBeInTheDocument();
    ont ete modifiees pour chercher le role d'accessibilite correct d'un onglet :
      expect(screen.getByRole('tab', { name: /hiragana/i })).toBeInTheDocument();
    La suite de tests passe desormais a 100% tout en validant la structure ARIA.

--------------------------------------------------------------------------------
ANOMALIE ID : B-003 | COMPOSANT : Frontend - Accessibilité du canvas de dessin
--------------------------------------------------------------------------------
*   Symptomes et description :
    Les tests d'accessibilite au clavier (a l'aide du lecteur d'ecran NVDA et de la
    navigation pas-a-pas par la touche 'Tab') ont revele que l'element '<canvas>'
    servant a dessiner les caracteres japonais etait completement ignore. L'utilisateur
    ne pouvait pas y acceder au clavier, et aucune alternative textuelle n'indiquait
    qu'il s'agissait d'une zone interactive de dessin pour l'apprentissage. De plus,
    il etait impossible d'effacer le tracé sans cliquer sur le bouton "Effacer".
*   Diagnostic et demarche de recherche :
    Par defaut, la balise HTML5 '<canvas>' est un element purement graphique et non
    interactif dans l'arbre d'accessibilite du navigateur (elle ne reçoit pas le focus).
    Il est necessaire de la declarer explicitement comme element interactif focalisable
    et de lier des evenements clavier pour reproduire les actions de la souris.
*   Resolution technique appliquee :
    Plusieurs modifications ont ete apportees dans 'Study.jsx' et 'index.css' :
    1. Ajout de l'attribut 'tabIndex={0}' sur l'element '<canvas>' pour l'inserer
       dans la boucle de navigation clavier.
    2. Ajout de l'attribut 'className="a11y-canvas"' associe a un style CSS globale
       ':focus-visible' qui applique une bordure violette néon tres contrastée.
    3. Ajout d'un attribut 'aria-label' dynamique decrivant la zone en fonction de la
       lettre active (ex: "Zone de dessin pour le caractere romaji 'a'. Appuyez sur
       Echap pour effacer").
    4. Ecriture de la fonction 'handleCanvasKeyDown' ecoutant la touche 'Escape' (Echap)
       pour vider instantanement le canvas.
    5. Ajout d'une zone d'annonce vocale dynamique 'aria-live="polite"' pour informer
       l'utilisateur de l'etat d'avancement des tracés.

ANOMALIE ID : B-004 | COMPOSANT : CI/CD - Depassement espace disque GitHub Actions
--------------------------------------------------------------------------------
*   Symptomes et description :
    Le pipeline d'integration continue configure sur GitHub Actions s'arretait au
    bout de 15 minutes en retournant une erreur critique de manque d'espace disque
    sur le runner virtuel (code d'erreur "No space left on device").
*   Diagnostic et demarche de recherche :
    En analysant les logs du build de notre conteneur de test, nous avons note que
    l'installation de la bibliotheque PyTorch (utilisee par Manga-OCR) telechargeait
    le paquet CUDA par defaut pour le support GPU (ce paquet represente plus de 4 Go
    de donnees compressées). Additionné aux dependances système et au cache de pip,
    le runner de GitHub Actions depassait sa limite stricte de stockage.
*   Resolution technique appliquee :
    Mise a jour du fichier '.github/workflows/backend-ci.yml' (et frontend-ci.yml) :
    1. Ajout d'une etape initiale de nettoyage du disque du runner virtuel pour liberer
       pres de 20 Go d'espace (suppression de l'environnement dotnet, des outils android
       et de la suite logicielle Haskell GHC).
    2. Configuration de l'environnement pip pour forcer l'installation de la version
       allegee de PyTorch CPU sans support CUDA :
       'PIP_EXTRA_INDEX_URL=https://download.pytorch.org/whl/cpu'
    3. Utilisation de l'option '--no-cache-dir' de pip pour eviter le stockage de
       fichiers d'installation temporaires volumineux.
    Le pipeline s'execute desormais en moins de 3 minutes de maniere stable.

4.3 Procedure de sauvegarde (Backup) et de restauration
--------------------------------------------------------------------------------
L'application stocke ses donnees de maniere locale dans le fichier 'manga-ocr/sensai.db'
et les mangas sous 'manga-ocr/uploads/'. Il est crucial de sauvegarder ces elements.

*   Script de sauvegarde automatique (A planifier via une tache cron) :
    Creez un script 'backup.sh' contenant :
      #!/bin/bash
      BACKUP_DIR="/var/backups/sensai"
      DATE=$(date +%F_%H%M%S)
      mkdir -p $BACKUP_DIR
      # Sauvegarde de la base de donnees SQLite
      cp /chemin/vers/Sensai/manga-ocr/sensai.db $BACKUP_DIR/sensai_db_$DATE.db
      # Sauvegarde des fichiers mangas importes
      tar -czf $BACKUP_DIR/sensai_uploads_$DATE.tar.gz -C /chemin/vers/Sensai/manga-ocr/uploads .
      # Conserver uniquement les 7 dernieres sauvegardes
      find $BACKUP_DIR -type f -mtime +7 -delete

*   Procedure de restauration en cas d'incident :
    1. Arreter l'application :
       docker compose down
    2. Restaurer le fichier de base de donnees :
       cp /var/backups/sensai/sensai_db_sauvegardee.db /chemin/vers/Sensai/manga-ocr/sensai.db
    3. Restaurer les medias :
       rm -rf /chemin/vers/Sensai/manga-ocr/uploads/*
       tar -xzf /var/backups/sensai/sensai_uploads_sauvegardees.tar.gz -C /chemin/vers/Sensai/manga-ocr/uploads
    4. Relancer l'application :
       docker compose up -d

PARTIE V : GUIDE D'UTILISATION DE L'APPLICATION
================================================================================

Ce guide vous accompagne pas-a-pas dans la prise en main de l'application SensAI
pour optimiser votre apprentissage de la langue japonaise.

--------------------------------------------------------------------------------
5.1 Importer vos mangas et configurer les pages
--------------------------------------------------------------------------------
Pour charger vos documents d'etude dans l'application :
1.  Connectez-vous a votre compte personnel SensAI.
2.  Sur votre tableau de bord, cliquez sur le bouton "📥 Importer" situe dans la
    barre de navigation superieure.
3.  Une fenetre de dialogue s'ouvre :
    - Cliquez sur la zone centrale pour selectionner un fichier PDF ou une image sur
      votre ordinateur.
    - Dans le champ "Titre personnalise", vous pouvez renommer le document (par exemple,
      saisir "One Piece Tome 1" a la place du nom de fichier brut complexe).
    - Sélectionnez le "Dossier de destination" si vous souhaitez le classer.
    - Si le fichier est un volumineux document PDF et que vous ne souhaitez etudier
      qu'un seul chapitre : cochez l'option "Extraire une plage de pages", puis
      entrez la plage (ex : pages 1 a 15).
4.  Cliquez sur "Importer". Le fichier est traite par le serveur et la couverture
    apparait instantanement dans votre bibliotheque de lecture.

--------------------------------------------------------------------------------
5.2 Utilisation du lecteur intelligent et de l'OCR
--------------------------------------------------------------------------------
1.  Cliquez sur la couverture de l'œuvre que vous souhaitez lire pour lancer le lecteur.
2.  Naviguez de page en page a l'aide des fleches laterales ou de votre souris.
3.  Lorsqu'une bulle de dialogue contient des kanji ou du vocabulaire inconnu :
    - Cliquez sur l'outil de selection (l'icone en forme de cadre de recadrage).
    - Cliquez et glissez votre curseur pour encadrer precisement le texte japonais
      a l'interieur de la bulle.
4.  Relachez la souris : l'application envoie l'image recadree au backend.
5.  En moins de deux secondes, un panneau d'analyse s'ouvre a droite de l'ecran :
    - Le texte japonais est retranscrit (OCR).
    - La traduction francaise adaptee au contexte de la scene s'affiche.
    - Le Romaji vous indique la prononciation correcte.
    - Un tableau lexical detaille chaque mot composant la phrase avec son type
      grammatical (nom, verbe, adjectif, particule).
6.  Pour retenir ce vocabulaire, cliquez sur le bouton "Ajouter aux flashcards".
    Le mot est sauvegarde dans vos dossiers de revisions.

--------------------------------------------------------------------------------
5.3 Etude et revisions SRS au clavier
--------------------------------------------------------------------------------
L'application integre un systeme de repetition espacee (SRS) base sur le celèbre
algorithme SM-2 pour ancrer le vocabulaire dans votre memoire a long terme.

1.  Dans la barre de navigation, cliquez sur "🧠 Revisions".
2.  Selectionnez le dossier de revision que vous souhaitez travailler (le nombre de
    cartes en attente d'etude y est indique).
3.  La premiere carte s'affiche (Recto).
4.  Si la carte est de type vocabulaire :
    - Tentez de vous souvenir de la traduction.
    - Appuyez sur la touche 'ENTREE' de votre clavier pour afficher le Verso.
    - Evaluez votre rappel en appuyant sur une des touches numeriques de votre clavier :
      * Touche '1' : Oubli total. La carte sera reinseree immediatement apres 3 fiches.
      * Touche '2' : Rappel difficile. La carte reviendra dans la meme session apres 6 fiches.
      * Touche '3' : Bon rappel. La carte reviendra dans 12 fiches pour validation.
      * Touche '4' : Rappel parfait. La carte est consideree comme acquise pour aujourd'hui.
5.  Si la carte est de type caractere (apprentissage du tracé d'un Kana ou Kanji) :
    - Le Romaji s'affiche a l'ecran (ex : "ka").
    - A l'aide de votre souris ou de votre stylet tactile, tracez le caractere
      japonais correspondant dans le grand cadre noir de dessin.
    - Si vous faites une erreur de tracé ou d'ordre des traits, le systeme vous en
      informe. Appuyez sur la touche 'ECHAP' de votre clavier pour effacer et recommencer.
    - Des que le caractere est correctement trace, le Verso apparait. Saisissez votre note
      SM-2 pour passer a la suite.
